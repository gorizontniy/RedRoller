# Дорожная карта Redroller

Внутренний документ для совместной разработки. Пользовательский вход в проект
остаётся в `README.md`; сюда можно складывать продуктовую кухню, очередность
работ, архитектурные решения и задачи для людей/агентов.

Redroller развивается из рулетки Yandex Cloud в единое локальное приложение для
нескольких хостеров: центр управления аккаунтами, общая панель запусков и
провайдерные парсеры/ротаторы.

Ближайший принцип: сначала доводим Yandex Cloud до стабильного эталона, потом
выносим общие контракты и добавляем новых провайдеров через эти контракты, а не
копированием текущего кода.

## Север проекта

Redroller должен стать desktop-приложением, где есть:

- общий центр аккаунтов всех провайдеров;
- локальное зашифрованное хранение ключей и токенов;
- единая панель парсинга/ротации;
- провайдерные формы настройки;
- провайдерные адаптеры под общим контрактом;
- общий журнал запусков, статусов и результатов;
- единый Windows release-flow;
- понятный GitHub-flow для людей и агентов.

Визуальное направление: операторская консоль. Плотная, индустриальная,
явная, без маркетингового лендинга. Пользователь должен сразу видеть аккаунты,
провайдеров, состояние запусков и безопасные действия.

## Провайдеры

| Провайдер | Статус | Цель |
|---|---|---|
| Yandex Cloud | Эталонная реализация | Стабилизировать и вынести из неё общий контракт. |
| Selectel | Запланирован | Добавить аккаунты, валидацию и skeleton парсера. |
| Timeweb Cloud | Запланирован | Добавить аккаунты, валидацию и skeleton парсера. |
| Cloud.ru | Запланирован | Добавить аккаунты, валидацию и skeleton парсера. |
| Остальные хостеры | Позже | Подключать только после стабилизации provider-контракта. |

## M0. Стабилизировать текущий Yandex Cloud baseline

Статус: release candidate.

Что должно остаться рабочим:

- аккаунты Yandex Cloud создаются и редактируются из web-панели;
- JSON-ключи и Telegram token хранятся зашифрованно;
- режимы “гибридная крутка” и “крутка 1 проекта” понятны в UI;
- целевые IP/CIDR редактируются отдельно от ключей и зон;
- Telegram-настройки сохраняются и тестируются;
- изоляция cloud-id и folder-id редактируется отдельно от аккаунта;
- успешный cloud/folder автоматически попадает в изоляцию;
- неподходящий успешный результат можно вручную удалить из истории: reserved address удаляется, disposable-folder отправляется на удаление, пустое managed disposable-cloud тоже отправляется на удаление, cloud/folder снимаются с изоляции, выбранный folder/cloud режима 1 проекта не удаляется;
- безопасный режим по умолчанию останавливается после первого найденного IP;
- режим сбора может продолжать поиск нескольких успешных IP;
- история IP даёт быструю ссылку в dashboard каталога Yandex Cloud;
- сборка `Redroller.exe` воспроизводится.

Критерии выхода:

- проходит команда:

```powershell
python -m unittest discover -s .\bin -p "test_*.py" -v
```

- новый пользователь может запустить Redroller, добавить Yandex-аккаунт и
  понять режимы без ручного редактирования JSON;
- live-действия не спрятаны за двусмысленными кнопками.
- `README.md` остаётся пользовательским гайдом без внутренних задач, agent-map
  и coordination notes.

## M0.1. Первый публичный релиз

Цель: подготовить чистый push/PR и релизную сборку без утечки локального runtime.

Чеклист:

- README проверен как инструкция для нового пользователя;
- `docs/ROADMAP.md`, `docs/AGENT_MAP.md`, `AGENTS.md` отражают текущие правила
  совместной разработки;
- PR описывает live-mode risk и результат тестов;
- перед push проверены `git status`, `git diff --check`, unit-тесты;
- в commit нет `.web-runtime`, SQLite, логов, ключей, локальных config и
  test-output;
- после merge можно собрать `Redroller.exe` через `bin/build_web_panel_exe.ps1`.

## M1. Общий центр аккаунтов

Цель: убрать жёсткую привязку приложения к полям Yandex Cloud.

Работы:

- добавить provider-id к аккаунтам;
- разделить общие поля аккаунта и provider-specific настройки;
- оставить шифрование секретов общим;
- показать карточки провайдеров в UI;
- мигрировать текущую SQLite-схему без потери Yandex-аккаунтов.

Критерии выхода:

- существующие Yandex-аккаунты мигрируют без потерь;
- UI умеет группировать аккаунты по провайдерам;
- Selectel, Timeweb Cloud и Cloud.ru могут быть показаны как placeholders без
  live-запусков.

## M2. Provider Adapter Contract

Цель: каждый хостер подключается через одинаковый backend-контракт.

Черновой контракт:

- `provider_id`;
- `display_name`;
- `credential_schema`;
- `validate_account`;
- `build_runtime_config`;
- `start_job`;
- `stop_job`;
- `read_status`;
- `sync_success_state`;
- `cleanup_or_isolate`.

Критерии выхода:

- Yandex Cloud работает через этот контракт;
- тесты проверяют контракт, а не только конкретные Yandex-поля;
- добавление нового провайдера не требует переписывать всю web-панель.

## M3. Единая панель парсинга

Цель: превратить текущую рулетку в provider-aware dashboard.

Работы:

- показывать активный provider, аккаунт, режим, цели, текущий IP/candidate и
  историю запусков;
- хранить run history с provider/account/job identifiers;
- добавить фильтры по провайдеру и аккаунту;
- оставить provider-specific детали доступными, но не шумящими по умолчанию.

Критерии выхода:

- Yandex-запуски выглядят и работают не хуже текущей версии;
- будущий Selectel/Timeweb запуск сможет появиться в той же истории;
- UI явно разделяет account center и parsing/run center.

## M4. Второй провайдер

Цель: доказать архитектуру на одном non-Yandex provider.

Первым лучше брать провайдера с простым API и минимально опасной cleanup-моделью.

Критерии выхода:

- аккаунт второго провайдера можно добавить в UI;
- parser умеет dry-run/demo mode;
- live mode имеет явное предупреждение и тесты;
- Yandex-термины не протекают в flow второго провайдера.

## M5. Release и совместная работа

Цель: сделать развитие проекта предсказуемым.

Правила:

- PR должен быть сфокусирован на одной теме;
- в PR указывать тесты или причину, почему они не запускались;
- не коммитить runtime, логи, базы, ключи и локальные конфиги;
- обновлять эту дорожную карту при изменении направления;
- обновлять `docs/AGENT_MAP.md` при изменении границ модулей;
- не выносить внутренние планы, agent instructions и PR-кухню в главный
  пользовательский README.
- не добавлять скрытую телеметрию: сбор успешных IP, usage stats или crash data
  допустим только через явное opt-in согласие пользователя.

## Пока не делаем

- облачный SaaS-control-plane;
- удалённую общую базу;
- multi-user auth;
- автоматические billing-решения между провайдерами;
- синхронизацию секретов между машинами;
- скрытый сбор успешных IP или другой телеметрии;
- новые provider-интеграции без тестов.
